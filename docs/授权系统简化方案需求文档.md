## 1. 方案概述

### 1.1 背景
当前系统采用离线激活码（HMAC/RSA签名 + wxid绑定）方式授权，存在以下问题：
- 签名体系复杂，需维护密钥/公钥
- 必须绑定wxid，用户操作繁琐
- 离线激活码易被复制传播
- 到期后强制停止，用户体验不佳

### 1.2 简化目标
采用**在线认证 + 离线使用**模式，简化授权流程：
1. 桌面端通过账号密码在线登录认证
2. 认证成功后返回授权到期时间，本地保存
3. 本地验证到期时间，离线使用至到期时间
4. 到期前自动提醒用户续费
5. 在线续费后自动更新到期时间

### 1.3 方案优势

| 对比维度 | 旧方案（离线激活码） | 新方案（在线认证） |
|---------|-------------------|------------------|
| **认证方式** | HMAC/RSA签名激活码 | 账号密码登录 |
| **绑定机制** | 必须绑定wxid | 无需绑定 |
| **签名体系** | 需维护密钥/公钥 | 无需签名体系 |
| **离线支持** | 完全离线 | 在线认证后离线使用 |
| **到期提醒** | 无 | 到期前3/7/15天提醒 |
| **续费方式** | 输入新激活码 | 在线支付自动更新 |
| **安全性** | 激活码可复制 | 账号密码+Token验证 |

## 2. 授权流程设计

### 2.1 整体流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户购买授权                              │
│              ↓                                                  │
│  用户在营销网站注册账号 → 购买套餐 → 支付成功 → 账号绑定授权       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      桌面端首次使用                              │
│              ↓                                                  │
│  桌面端启动 → 无本地授权 → 显示登录界面 → 输入账号密码             │
│              ↓                                                  │
│  在线验证账号密码 → 返回授权信息（到期时间） → 本地保存             │
│              ↓                                                  │
│  验证到期时间未过 → 进入主界面 → 正常使用                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      桌面端日常使用                              │
│              ↓                                                  │
│  桌面端启动 → 读取本地授权 → 检查到期时间                          │
│              ↓                                                  │
│  到期时间未过 → 进入主界面（显示到期倒计时）                        │
│  到期时间已过 → 显示登录界面，要求重新认证                         │
│              ↓                                                  │
│  到期前3/7/15天 → 桌面端弹出提醒 → 引导在线续费                   │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                        续费流程                                  │
│              ↓                                                  │
│  用户点击续费 → 跳转营销网站 → 选择套餐 → 支付成功                 │
│              ↓                                                  │
│  桌面端检测到续费 → 自动更新本地到期时间 → 延长使用期限            │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 认证流程详细设计

#### 2.2.1 首次登录认证

**流程**:
1. 用户在桌面端登录界面输入邮箱和密码
2. 桌面端调用后端 `POST /api/auth/login` 接口
3. 后端验证账号密码，返回 JWT Token 和授权信息
4. 桌面端保存 Token 和授权到期时间到本地
5. 验证到期时间未过，进入主界面

**接口**:
```
POST /api/auth/login
请求体: {
    "email": "user@example.com",
    "password": "123456"
}
响应体: {
    "success": true,
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "user": {
        "id": 1,
        "email": "user@example.com",
        "username": "张三"
    },
    "license": {
        "expires_at": "2026-12-31T23:59:59",
        "subscription_type": "yearly",
        "max_groups": 10
    }
}
```

#### 2.2.2 本地授权验证

**流程**:
1. 桌面端启动时读取本地保存的授权信息
2. 检查当前时间是否在到期时间之前
3. 如果未到期，进入主界面并显示到期倒计时
4. 如果已到期，强制要求重新登录认证

**本地存储结构**:
```json
{
  "auth_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "user_id": 1,
  "email": "user@example.com",
  "license": {
    "expires_at": "2026-12-31T23:59:59",
    "subscription_type": "yearly",
    "max_groups": 10,
    "last_sync_at": "2026-06-16T10:00:00"
  }
}
```

#### 2.2.3 到期提醒机制

**触发时机**:
- 距离到期还有 **15天**：轻量提醒（右上角小图标）
- 距离到期还有 **7天**：弹窗提醒（可关闭）
- 距离到期还有 **3天**：每日弹窗提醒（必须确认）
- 距离到期还有 **0天**：进入只读模式，无法处理消息

**提醒内容**:
```
"您的授权即将到期，剩余 X 天。请前往营销网站续费。"
[去续费] [稍后提醒]
```

#### 2.2.4 在线续费流程

**流程**:
1. 用户在桌面端点击"续费"按钮
2. 自动打开浏览器跳转至营销网站续费页面（自动携带用户信息）
3. 用户选择套餐并完成支付
4. 支付成功后，营销网站调用后端 API 更新用户授权到期时间
5. 桌面端定期（每1小时）检查授权状态
6. 检测到到期时间更新后，自动刷新本地授权信息

**后端接口**:
```
POST /api/license/renew
请求体: {
    "user_id": 1,
    "plan": "yearly",
    "payment_method": "alipay",
    "transaction_id": "2026061622001234567890"
}
响应体: {
    "success": true,
    "license": {
        "expires_at": "2027-12-31T23:59:59",
        "subscription_type": "yearly"
    }
}
```

## 4. API接口设计

### 4.1 认证接口

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/auth/register | POST | 用户注册 |
| /api/auth/login | POST | 用户登录（返回授权信息） |
| /api/auth/me | GET | 获取当前用户信息（含授权状态） |

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/license/info | GET | 获取当前用户授权信息 |
| /api/license/renew | POST | 续费授权 |
| /api/license/extend | POST | 延期授权（管理员） |
| /api/license/check | GET | 检查授权状态（供桌面端定期调用） |

### 4.3 订单接口

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/orders | GET | 获取用户订单列表 |

| /api/orders/:id | GET | 获取订单详情 |
| /api/orders/:id/pay | POST | 支付订单 |

## 7. 安全设计

### 7.1 认证安全

- JWT Token 有效期：24小时
- Token 存储：桌面端本地加密存储
- 密码加密：bcrypt 哈希（10轮）
- 登录尝试限制：5次/分钟，超过后锁定15分钟

### 7.2 传输安全

- API 请求必须使用 HTTPS
- 请求体加密传输
- 防止中间人攻击

### 7.3 数据安全

- 用户密码不明文存储
- 敏感信息脱敏显示
- 定期数据备份

### 7.4 防篡改

- 本地授权信息包含校验和
- 检测到篡改时强制重新认证

## 10. 注意事项

1. **向后兼容**: 现有离线激活码用户需要平滑迁移到新系统
2. **网络容错**: 网络不可用时使用本地缓存的授权信息
3. **时间同步**: 桌面端需定期同步服务器时间，防止本地时间篡改
4. **数据备份**: 定期备份本地授权信息，防止数据丢失
5. **隐私保护**: 遵守相关隐私法规，不收集不必要的用户信息
