# 行业插件开发规范文档

## 1. 概述

### 1.1 文档目的
本文档定义 TonjClaw 系统中行业插件的开发规范和接口标准，确保各行业插件与主系统的无缝集成，同时保证插件具备良好的可扩展性、兼容性和可维护性。

### 1.2 设计原则

| 原则 | 描述 |
|------|------|
| **模块化** | 每个行业插件作为独立模块，拥有自己的数据模型、服务层和API |
| **松耦合** | 插件与主系统通过标准接口交互，减少依赖 |
| **可插拔** | 支持独立安装、更新、卸载，不影响其他插件和主系统 |
| **可扩展** | 提供统一的扩展点，支持自定义功能和业务逻辑 |
| **一致性** | 遵循统一的代码规范、数据格式和API风格 |
| **版本化** | 支持插件版本管理、升级和回滚 |
| **权限化** | 插件权限声明与用户权限检查机制 |
| **数据隔离** | 行业插件数据独立存储，实现数据隔离 |

### 4.3 数据模型规范

所有行业插件的数据模型必须遵循以下规范：

1. **表名前缀**: `t_{industry}_`，如 `t_seafood_order`、`t_construction_worker`
2. **字段命名**: 使用下划线分隔，全小写
3. **时间戳**: 必须包含 `created_at` 和 `updated_at` 字段
4. **状态字段**: 使用 `is_active` 字段标记记录状态
5. **软删除**: 推荐使用 `is_deleted` 字段实现软删除
6. **用户隔离**: 必须包含 `user_id` 字段实现数据隔离

### 4.4 版本管理标准

#### 4.4.1 版本号格式
采用语义化版本控制（Semantic Versioning）：
- 格式: `MAJOR.MINOR.PATCH`
- 示例: `1.0.0`, `1.1.0`, `1.0.1`

#### 4.4.2 版本升级规则
| 类型 | 条件 | 示例 |
|------|------|------|
| **MAJOR** | 不兼容的API变更 | `1.0.0` → `2.0.0` |
| **MINOR** | 向后兼容的功能新增 | `1.0.0` → `1.1.0` |
| **PATCH** | 向后兼容的问题修复 | `1.0.0` → `1.0.1` |

#### 4.4.3 版本比较规则
- 使用 `services/plugin_service.py` 中的 `compare_versions()` 方法
- 返回值: -1（v1 < v2）、0（相等）、1（v1 > v2）

#### 4.4.4 版本约束格式
支持以下版本约束：
| 约束 | 说明 | 示例 |
|------|------|------|
| `>=` | 大于等于 | `>= 1.0.0` |
| `<=` | 小于等于 | `<= 2.0.0` |
| `>` | 大于 | `> 1.0.0` |
| `<` | 小于 | `< 2.0.0` |
| `~=` | 兼容版本 | `~= 1.0` |

### 4.5 依赖管理标准

#### 4.5.1 依赖声明格式
```json
[
    {"plugin_code": "core", "min_version": "1.0.0"},
    {"plugin_code": "wechat", "min_version": "1.0.0"}
]
```

#### 4.5.2 循环依赖检测
系统会自动检测循环依赖，例如：
- 插件A → 插件B → 插件C → 插件A（循环依赖，禁止安装）

#### 4.5.3 依赖解析顺序
依赖按拓扑排序后的顺序安装，确保前置依赖先于依赖者安装。

## 5. 插件开发规范

### 5.1 代码规范

#### 5.1.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 模块名 | 小写+下划线 | `construction_service.py` |
| 类名 | PascalCase | `ConstructionService` |
| 方法名 | 小写+下划线 | `check_in_worker` |
| 变量名 | 小写+下划线 | `worker_name` |
| 常量名 | 全大写+下划线 | `MAX_RETRY_COUNT` |

#### 5.1.2 文档规范

- 每个模块必须有模块级文档字符串
- 每个类必须有类级文档字符串
- 每个公共方法必须有方法级文档字符串
- 文档字符串使用Google风格

#### 5.1.3 错误处理

- 使用统一的错误返回格式: `{'success': False, 'error': '错误信息'}`
- 捕获异常时记录日志
- 对外接口使用try-except包装

### 5.2 数据格式规范

#### 5.2.1 API响应格式

```json
{
  "success": true,
  "code": 200,
  "message": "操作成功",
  "data": {},
  "count": 0,
  "timestamp": "2024-01-15T10:30:00Z"
}
```

#### 5.2.2 规则文件格式

```json
{
  "version": "1.0",
  "industry": "餐饮配送",
  "parse_rules": [],
  "stat_rules": [],
  "reply_rules": []
}
```

### 5.3 安全规范

1. **输入验证**: 所有用户输入必须进行验证
2. **SQL注入防护**: 使用ORM参数化查询
3. **XSS防护**: 输出内容进行HTML转义
4. **权限控制**: 使用 `@login_required` 装饰器
5. **敏感信息**: 不记录密码等敏感信息到日志

## 11. 插件开发流程

### 10.1 开发步骤

```
1. 需求分析 → 2. 数据模型设计 → 3. 服务层开发 → 4. API接口开发
        ↓
5. 规则配置 → 6. 测试验证 → 7. 打包发布 → 8. 文档编写
```

### 10.2 测试规范

1. **单元测试**: 覆盖核心业务逻辑
2. **集成测试**: 测试插件与主系统的集成
3. **性能测试**: 确保响应时间符合要求
4. **兼容性测试**: 测试不同版本的兼容性
5. **权限测试**: 验证权限控制机制

### 10.3 发布流程

1. **代码审查**: 提交代码进行审查
2. **自动测试**: CI/CD自动运行测试
3. **版本发布**: 发布到插件市场
4. **更新通知**: 通知已安装用户
